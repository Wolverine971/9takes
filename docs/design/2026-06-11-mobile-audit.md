# 9takes Mobile Audit — post-design-push

> **Date:** 2026-06-11
> **Scope:** Mobile (360–390px) pass over every user-facing surface, following the 2026-06-09→11 design push (`docs/design/2026-06-09-design-audit.md`). Code-level sweep of global chrome, home/marketing, Q&A, blog surfaces, auth/search/misc + a live browser pass at 390×844 and 360×740 (dark + light) with programmatic overflow checks on every page.
> **Touch-target stance (ratified by DJ):** density first, Instagram-style. Targets are NOT flagged for being under 44px; only genuinely hard-to-hit cases (sub-32px crowded by neighbors, or tiny destructive actions) are reported.
> **Screenshots:** `/tmp/9takes-mobile-audit/` (50 files — ephemeral, copy if you want to keep them).

---

## Verdict

**Mobile fundamentals are strong — and the audit caught the site being down.** Zero horizontal overflow on every page tested at both 390 and 360 (rare for a content site), zero console errors, light mode fully functional, and the homepage hits exactly the dense-but-clean register you're aiming for. But two production-grade bugs were live when the audit started:

1. **Every `/questions/[slug]` page returned HTTP 500 in production** (verified against three live slugs on 9takes.com) — an SSR crash introduced by the 2026-06-10 clamp() conversion. **Fixed in the working tree; deploy ASAP.**
2. **Form inputs could render invisible site-wide** — yesterday's Phase-7 token deletion collapsed the global input fill and border to the same `--stone-warm`, so on stone-warm cards (login) the fields disappeared entirely. **Fixed in the working tree.**

Below that headline: the mobile experience is genuinely good, with a short list of real issues concentrated in (a) iOS-specific viewport/keyboard behavior, (b) a few sub-16px text regressions that undercut the prose-measure work, and (c) surfaces the V5 sweep never reached (search page, book-session, error page).

## 🔴 Fixed during this audit (in working tree — needs commit + deploy)

### 1. Production down: all question detail pages 500

`src/lib/components/questions/QuestionDisplay.svelte` — the `$derived` computing `fontSize` sat **above** the `const SIZE_TIERS` it depends on. `$derived` evaluates eagerly during SSR, so every server render threw `ReferenceError: Cannot access 'SIZE_TIERS' before initialization`. On the client `$derived` is lazy, which is why it never showed locally in the browser. Verified 500 on production for `how-do-you-handle-stress`, `belief-changed-last-3-years-changed`, `best-parenting-advice-ever-recieved`.

**Fix applied:** moved the derived below the declarations (with a comment explaining the SSR eagerness). Both test slugs return 200 locally.
**Lesson:** in Svelte 5, treat `$derived` like a live expression at declaration position during SSR — dependencies must be declared above it.

### 2. Global inputs invisible (Phase-7 regression)

`src/scss/index.scss:726-727` — the global `input[type=…]/select` rule had `background-color: var(--stone-warm); border: 1px solid var(--stone-warm)` (border = its own fill). Pre-Phase-7 this was `--bg-elevated` fill + `--bg-highlight` border (two different grays); the alias deletion mapped both to `--stone-warm`. On the login card (also `--stone-warm`) the email/password fields were literally invisible — confirmed in live screenshots.

**Fix applied:** fill `--stone-mid`, border `--stone-edge` (the V5 equivalent of the old elevated+highlight pair). Verified visible in browser after fix.
**Residual cleanup (low):** `input[type="email"]` (0,1,1) ties Svelte-scoped `input.s-hash` (0,1,1), so source order decides which wins — in dev this is racy. Long-term, drop the global rule's specificity (wrap in `:where()`) so component styles always win.

## What you're doing well

- **Zero horizontal overflow anywhere.** Every page passed `scrollWidth <= innerWidth` at 390 and 360. The `overflow-x: clip` backstop with `@supports` fallback (`+layout.svelte:661-673`) plus `minmax(0, 1fr)` grid discipline is working.
- **The homepage is the density benchmark you want.** The 9-type primer table, locked-take cards, case-file grid, and corpus stats stack tight with no wasted air, and still fit cleanly at 360 (`19-home-360-table.png`). The library grid's mobile treatment (2-across, square crops, 15px clamped names, hover-transform disabled on touch, mobile-only type chip — `+page.svelte:1943-2127`) is exemplary.
- **CSS-only responsive switching where it was promised.** Header renders both layouts and lets media queries pick (no hydration swap); QuestionDisplay's length-tiered `clamp()` SSRs the correct size at every width (now that it actually SSRs at all).
- **The iOS zoom guard is well-architected** — `:where(input, select, textarea) { font-size: 16px }` at zero specificity (`index.scss`), reinforced explicitly in HeaderSearch, Interact's composer grid, account, and book-session. (Two component-level violations found, below.)
- **CLS discipline on content cards.** CaseCard and RelatedPosts images ship explicit `width`/`height` + `aspect-ratio`; titles line-clamp instead of overflowing; RelatedPosts forces its hover overlay visible on touch.
- **Tables in `.blog` articles get the correct mobile recipe** (`display: block; overflow-x: auto; -webkit-overflow-scrolling: touch` — `blog.scss:1239`), and TOC links wrap with `overflow-wrap: anywhere` instead of truncating.
- **Comment actions are touch-first**: Like/Reply rows at `min-h-11`, settings behind an explicit 44px trigger, destructive Flag behind popover + confirm modal with full-width stacked buttons on mobile.
- **Light mode holds up on mobile** — consistent amber-on-cream, no broken body contrast on home or articles.
- **Dossier pages read beautifully on a phone**: prose measure, meta row, 2-up similar profiles, reachable comment box (`11-pa-dossier-*.png`).
- **Modal scroll-lock uses a counter** so stacked modals don't unlock the body early; global `prefers-reduced-motion` guard exists.

## What you're not doing well (patterns, not one-offs)

1. **iOS viewport units and safe areas are unhandled.** `100vh` everywhere (`MobileNavNew` panel, Modal full-mobile, bottom-sheet), no `dvh`/`svh`, no `env(safe-area-inset-*)` anywhere. Consequence: the mobile nav's **Login/Register actions sit behind the iOS bottom toolbar**, full-screen modals have dead/hidden zones.
2. **Sub-16px text keeps reappearing.** The 2026-06-09 audit killed mobile sub-16 paragraphs in `blog.scss`, but the same pattern survives in four places (global `p { 0.8rem }`, enneagram-corner articles, comment composer, FAQ at ≤380px). The system needs a lint or a single mobile-typography source instead of per-file overrides.
3. **The V5 sweep has unreached pockets.** `/search` (mint-gradient CTA with raw hex, glass-morphism shells, raw blue/pink/yellow badges), `/book-session` (8 glass panels + `backdrop-filter: blur(14px)`), `+error.svelte` (completely unstyled scaffolding with placeholder copy), Email-Signup and BlogPurpose (both white-on-amber — the exact contrast bug purged from Comment.svelte).
4. **JS width-branching wasn't fully eradicated.** QuestionContent tabs (`innerWidth > 575`), SearchQuestion (`isMobile` set once in `$effect`, SSR defaults desktop), QuestionItem date format — all still hydration-swap. Header/QuestionDisplay got fixed; the pattern didn't.
5. **Two scroll-lock owners fight.** MobileNavNew and Modal both write `document.body.style.overflow` directly; closing one while the other is open unlocks the page behind it.
6. **Legacy-token fallbacks linger in shared atoms.** ComboBox styles against retired vars with raw fallbacks (and `rgba(var(--lamp-glow), 0.1)` — invalid CSS, so its focus ring/selected states silently don't render); SearchQuestion fights it with nine `!important` overrides.
